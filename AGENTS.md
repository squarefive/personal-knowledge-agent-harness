# Agents

本文定义本项目第一版 Agent 的职责边界、行为原则和工具协作方式。

## Agent 定位

本项目只有一个 Agent：本地个人 Q&A 知识库 Agent。

它的任务不是陪聊，也不是替用户维护完整 Wiki，而是帮助用户把零散 Q&A 变成可检索、可追溯、可复用的本地知识资产。

## 核心分工

### 模型负责判断和表达

模型负责：

- 判断用户意图是录入知识还是提问
- 将用户录入的 Q&A 整理成结构化知识卡片
- 提取摘要、关键词和可检索信息
- 根据检索结果组织回答
- 判断证据是否足够
- 在依据不足时拒绝编造

### 工具负责执行动作

工具负责：

- 写入 SQLite
- 查询 SQLite
- 更新知识卡片
- 返回结构化检索结果
- 提供可追溯的来源信息

模型不应假装已经保存、查询或更新数据。凡是涉及长期记忆的动作，都必须通过工具完成。

### SQLite 负责长期记忆

SQLite 是第一版唯一长期记忆来源。

模型上下文只能作为短期工作区，不能被当作可靠记忆。需要长期保留的知识必须落库。

## 第一版能力

### 录入 Q&A

当用户提供一组 Q&A，Agent 应该：

1. 保留原始问题和原始答案
2. 整理出简明 summary
3. 提取 keywords
4. 生成结构化知识卡片
5. 调用工具保存到 SQLite
6. 向用户确认保存结果和来源标识

### 回答问题

当用户提出问题，Agent 应该：

1. 调用工具检索 SQLite
2. 判断检索结果是否相关且足够
3. 基于检索结果回答
4. 附带来源
5. 如果没有依据，明确说明无法从本地知识库回答

## 回答格式建议

有依据时：

```text
回答：
...

来源：
- card_id: ...
  原始问题: ...
```

依据不足时：

```text
本地知识库中没有找到足够依据回答这个问题。

你可以补充一条 Q&A，我会把它整理并保存为知识卡片。
```

## 禁止行为

Agent 不得：

- 在没有检索本地知识的情况下声称“根据知识库”
- 编造不存在的来源
- 把模型猜测写成事实
- 把临时上下文当作长期记忆
- 自动扩展到 Wiki、文件监听、周报或多 Agent
- 为了回答完整而引入无来源外部知识

## 来源要求

每个回答都应能追溯到至少一个知识卡片。

来源至少包含：

- card_id
- 原始问题 question
- 来源类型 source_type
- 创建时间 created_at

如果实现阶段需要更强追溯能力，可以增加 source_ref、version 或 raw_input 字段。

## 设计判断

第一版应优先保证闭环真实存在，而不是追求复杂智能。

更重要的是：

- 能保存
- 能检索
- 能回答
- 能引用来源
- 不能回答时能诚实说明

只有这个闭环稳定后，才考虑扩展更多知识管理能力。

## 本地开发规约索引

AI Coding 工具在本仓库工作时，必须读取并遵守以下本地规约：

- `docs/guidelines/collaboration-preferences.md`: 用户协作偏好，包括文档修改计划、代码修改计划、伪代码要求和提交偏好。
- `docs/guidelines/ai-coding-behavior.md`: AI Coding 行为规约，包括先澄清再编码、简单优先、外科手术式修改和目标驱动验证。
- `docs/templates/agent-development-context.template.md`: 具体 Agent 开发上下文模板。

涉及具体 Agent 的新增、修改或重构时，还必须读取对应的 `docs/agents/{{module_name}}.md`。

新增或修改 Agent 开发上下文模板，或新增、修改具体 Agent 开发上下文文档后，必须运行 `scripts/check-agent-doc-format.py` 检查模板与具体 Agent 文档格式；检查通过并提交文档后，才能开始对应代码实现。

如果 Agent 开发上下文模板是有意升级，必须同步更新 `scripts/check-agent-doc-format.py` 中的模板 hash 常量，并在同一次文档提交中锁定新的模板版本。

如果实现前发现具体 Agent 文档与目标需求不一致，或需要调整 Agent 的角色边界、工具契约、数据模型、核心流程、外部依赖、权限规则等设计内容，必须先修改对应 Agent 文档，并将文档变更提交到本地 Git；在文档版本被锁定后，才能开始代码实现。
