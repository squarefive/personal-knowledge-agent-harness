# AI Coding 行为规约

本文档用于减少 AI Coding 工具常见编码错误。使用时应与项目级说明、Agent 开发上下文文档和用户协作偏好合并遵守。

权衡：本文档偏向谨慎而不是速度。对于低风险的简单任务，可以在明确假设后快速处理。

## 1. 先想清楚再写代码

不要假设，不要隐藏困惑，要显式说明取舍。

实现前应：

- 明确说明假设。
- 如果存在多个解释，先列出差异，不能静默选择。
- 如果存在更简单方案，应主动说明。
- 如果需求不清、风险较高或会影响架构 / 数据 / 权限，应先停下来提问。

## 2. 简单优先

用能解决问题的最小代码，不做投机式扩展。

要求：

- 不实现用户没有要求的功能。
- 不为单次使用代码抽象新层。
- 不添加未被要求的灵活性、配置项或扩展点。
- 不为未发生、未声明、不可触达的场景添加复杂防御。
- 不能省略必要的边界校验和失败反馈。
- 如果实现明显可以更短、更直接，应先简化。

自检问题：

> 资深工程师会不会认为这段实现过度复杂？

如果答案是会，应简化。

## 3. 外科手术式修改

只改必须修改的内容，只清理自己造成的问题。

编辑现有代码时：

- 不顺手改进相邻代码、注释或格式。
- 不重构没有坏掉的代码。
- 匹配项目现有风格，即使个人偏好不同。
- 发现无关死代码时，只说明，不擅自删除。

当本次修改造成未使用内容时：

- 移除由本次修改导致的未使用 import、变量、函数。
- 不删除修改前已经存在的死代码，除非用户明确要求。

判断标准：

> 每一行变更都应该能追溯到用户请求。

## 4. 目标驱动执行

把任务转换成可验证目标，并循环到验证完成。

示例：

- “添加校验”应转换为：为非法输入写测试，再让测试通过。
- “修复 bug”应转换为：先写出能复现 bug 的测试，再修复。
- “重构 X”应转换为：确认重构前后测试都通过。

多步骤任务计划应包含验证方式：

```text
1. [步骤] -> verify: [检查方式]
2. [步骤] -> verify: [检查方式]
3. [步骤] -> verify: [检查方式]
```

强成功标准可以支持独立推进。弱标准，例如“让它能用”，需要先澄清。

## 5. 生效判断

这些规约生效时，应体现为：

- diff 中不必要变更减少。
- 因过度复杂导致的返工减少。
- 澄清问题发生在实现前，而不是错误发生后。
