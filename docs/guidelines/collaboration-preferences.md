# 用户协作偏好

本文档记录本仓库中的用户协作偏好。AI Coding 工具在进行本地文档修改、代码修改、计划制定和提交前，应优先遵守本文档。

## 1. 计划先行

除非用户明确要求执行修改，否则 AI Coding 工具不得立即修改本地文件、运行会产生持久化变更的命令或提交 Git。

### 1.1 计划必填项

在给出计划时，必须先列出：

- 工作区
- Git 分支
- 背景
- 目标
- 修改范围

### 1.2 修改范围格式

在给出计划前，必须先完成必要调研，形成确定性的变更文件范围结论。

计划开头必须使用固定模板汇总本次准备新增、修改或删除的文件路径。修改范围只用于统观全局，每个文件必须且只需包含文件路径和一句话修改说明，不展开具体文案、demo、伪代码或实现细节。

修改范围必须按以下格式列出：

```markdown
修改：
- `path/to/file`: 一句话说明修改内容。

新增：
- `path/to/file`: 一句话说明新增内容。

删除：
- `path/to/file`: 一句话说明删除原因。
```

如果某一类没有内容，必须写 `无`，例如：

```markdown
新增：无
删除：无
```

该文件范围必须是调研后的确定性结论，不能用“可能”“预计”“大概”等不确定表达替代。如果调研后仍无法确定完整文件范围，必须明确说明不确定点、原因和需要用户确认的事项，不得把不确定范围包装成确定计划。

### 1.3 实施计划展开

具体文档写入内容、Markdown 示例、代码伪代码、逻辑草稿和验证方式，必须放在后续实施计划中按步骤展开。

涉及本地文档修改时，实施计划中必须明确列出准备写入或修改的具体内容，不能只描述修改思路。实施计划应包含准备新增、替换或删除的章节，以及拟写入的核心文本或完整片段。

涉及代码修改时，实施计划中必须先单独列出修改文件范围。如果某一步包含代码修改，必须先给出该步骤对应的伪代码或逻辑草稿，再开始实际修改代码。

### 1.4 分支策略

计划也是变更控制的一部分。开始修改前，必须先确认当前 Git 分支和工作区状态。

禁止直接在 `main` 或 `master` 上进行新增、修改或删除文件的操作，除非用户明确要求在主分支上修改。

如果当前处于 `main` 或 `master`，且任务会新增、修改或删除文件，必须先基于当前分支创建任务分支，再开始文件修改。

如果当前已经处于非 `main` / `master` 的任务分支，且新任务与当前分支方向一致，可以继续在当前分支完成，不需要重复新建分支。

如果新任务与当前分支方向不一致，必须基于 `main` 或用户指定的基线分支新建任务分支，不得把不同方向的修改混入同一分支。

新分支命名应简短表达任务意图，例如 `docs/update-collaboration-preferences`、`feat/local-qa-agent` 或 `fix/agent-doc-format-check`。

如果当前工作区存在未提交变更，必须先说明变更状态、判断是否与新任务同方向，并等待用户确认，不得直接切分支或混入新任务。

### 1.5 合并策略

当用户要求将任务分支合入 `main` / `master` 时，默认保留明确的 merge commit，使用：

```bash
git merge --no-ff <branch>
```

除非用户明确要求 fast-forward，否则不得使用 fast-forward merge 合入主分支。

如果任务分支落后于 `main` / `master`，应先切到任务分支并合并最新主分支，例如：

```bash
git switch <branch>
git merge main
```

在任务分支上解决冲突并完成必要验证后，再切回主分支执行 `git merge --no-ff <branch>`。

不要默认 rebase 任务分支；仅当用户明确要求时才使用 rebase。

合并后不要默认删除任务分支，除非用户明确要求清理。

## 2. 执行确认

只有当用户明确表达执行意图时，才能开始修改本地文件、运行会产生持久化变更的命令或提交 Git。

明确执行意图包括：

- “执行”
- “开始修改”
- “落地”
- “按计划执行”
- “提交”
- 其他清晰表达允许修改或提交的指令

如果用户只是在讨论方案、询问看法、要求计划、要求核对范围，不得执行修改。

## 3. 提交偏好

如果用户要求提交，提交前必须检查实际 diff，并基于实际变更生成提交信息。

文档设计变更和对应代码实现变更应拆分提交，除非只是修正文档中的错别字、路径或示例。
